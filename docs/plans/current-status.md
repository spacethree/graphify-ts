# graphify-ts current status and roadmap

This document keeps the implementation status, scope notes, limitations, and next-step planning details that were moved out of the top-level `README.md` so the README can stay end-user focused.

## Current status

Implemented in this repository:

- project scaffold with strict TypeScript + Vitest
- extraction schema contracts and validation
- additive schema-v2 extraction metadata contracts for `schema_version`, layered `base` / `semantic` / `media` tagging, provenance validation on nodes / edges / hyperedges, and fixture-backed regression coverage for legacy-v1 compatibility
- schema normalization helpers that upgrade legacy extraction payloads immutably with default schema-version / layer / provenance metadata and apply those defaults during graph build
- cache utilities with atomic writes
- corpus/file detection with `.graphifyignore`, hidden-file skipping, symlink controls, incremental manifest helpers, and explicit `graphify-out/memory/` note inclusion without re-ingesting generated graph artifacts
- graph assembly from extraction JSON
- shared extractor helpers now split under `src/pipeline/extract/` to keep the main pipeline entry point smaller and more modular
- extraction MVP for:
  - Python, Go, Java, and Ruby via portable `web-tree-sitter` + `@vscode/tree-sitter-wasm` parsing for owner-aware type/method extraction, imports, async Python function support, rationale attachment, and common call-graph edges, including Ruby command-style call resolution plus singleton/constant-receiver dispatch for common fixture-backed cases
  - Lua and `.toc` addon manifests via lightweight source/metadata scanners
  - Elixir and Julia via lightweight end-delimited source scanners
  - PowerShell and Objective-C via lightweight platform-aware source scanners
  - JavaScript / TypeScript via the TypeScript compiler API, including static imports, re-exports, `import = require(...)`, CommonJS `require(...)`, dynamic `import()`, and direct call-graph coverage
  - remaining brace-style languages (currently including Kotlin, Scala, C#, Rust, Swift, PHP, Zig, and C/C++-style signatures) via a portable regex-based structural extractor with common inheritance/conformance support, including multiline Scala/Kotlin/Swift signature coverage from the fixture-backed parity cases
  - Markdown / text / RST documents via deterministic section parsing, frontmatter metadata lifting (`title`, `source_url`, `captured_at`, `author`, `contributor`, `source_nodes`, etc.), explicit raw ingest-provenance emission for registry-driven text captures, normalization-time ingest-provenance fallback/dedupe for backward compatibility, source-node reference edges, local-reference detection, DOI/arXiv/LaTeX-citation extraction, numbered-citation resolution, and bibliography-reference nodes enriched with local reference metadata plus explicit bibliography `source_url` lifting when plain external URLs are present
  - DOCX documents via synchronous zip/XML title/author/subject/description lifting, heading extraction, local citation detection, and basic numbered-reference parsing inside detected references sections
  - XLSX spreadsheets via synchronous zip/XML workbook metadata, sheet-name extraction, and shared-string citation detection
  - text-like paper corpora plus heuristic PDF title/author/subject extraction, `Tj`/`TJ` text recovery, inline citation detection, and basic numbered-reference extraction when recovered text includes local bibliography entries
  - image assets as first-class graph nodes
- sidecar-backed non-code extraction now fans out explicit raw ingest provenance across PDF/image/DOCX/XLSX nodes and edges (including file-only DOCX fallback when hidden sidecar metadata is present), while normalization remains the backward-compatible fallback/dedupe path for older payloads
- cache-backed multi-file extraction for the implemented corpus types, including extractor-cache version invalidation when raw extraction metadata contracts change so stale pre-metadata payloads do not leak back through `extract()`
- bridge-aware community clustering and cohesion scoring

## Known limitations

- Python extraction now runs through the same portable WASM tree-sitter path as Go and Java. The current AST-backed slice covers the fixture-backed structural, async-function, rationale, and common cross-file `from ... import ...` parity cases already under test, while deeper language-specific edge cases still remain lighter than a dedicated full-reference parser stack.
- Ruby extraction now runs through the same portable WASM tree-sitter path as Python, Go, and Java. The current AST-backed slice covers common `module` / `class` / `def` structure, `require` / `require_relative` imports, inheritance, direct calls, command-style no-paren calls, and common singleton/constant-receiver dispatch already covered by fixture-backed tests, while metaprogramming-heavy Ruby edge cases still remain lighter than a dedicated full-reference parser stack.
- Lua, `.toc`, Elixir, Julia, PowerShell, and Objective-C extraction now cover common module/class/function or metadata structure plus straightforward imports, references, and call edges, but they remain lightweight scanners aimed at common source patterns rather than full language parsers.
- JavaScript / TypeScript extraction now covers classes, interfaces, heritage clauses (`extends` / `implements`), methods, nested function declarations/closures, top-level and class-field arrow/function-expression bindings, static imports, re-exports, `import = require(...)`, CommonJS `require(...)`, dynamic `import()` usage, and direct call relationships. More advanced decorator-heavy metadata and some uncommon AST edge cases are still lighter than a fuller parser-backed implementation.
- Go and Java extraction now run through the same portable WASM tree-sitter path used for Python and Ruby. The current AST-backed slice focuses on common type, interface, record, method, import, and call-graph patterns already covered by tests; deeper language-specific edge cases still need more parity work.
- The remaining portable multi-language extractor is intentionally heuristic. It now emits useful structure for several brace-style languages, including owner-aware method extraction for common `impl` / `extension` blocks, common inheritance/conformance clauses, Kotlin / Scala expression-bodied definitions, Swift `func ... -> ...` signatures, Zig `@import`/`struct`/`fn` patterns, and qualified `Type::method` definitions, but it is still regex-based rather than AST-backed, so deeply nested signatures, macros, or language-specific edge cases can still be missed.
- Document and text-like paper extraction is still deterministic rather than generative, but it now models headings, containment, markdown links, lifted frontmatter/query-memory metadata, explicit raw ingest provenance for registry-driven text captures with normalization-time fallback/dedupe for backward compatibility, source-node references, local filename mentions, DOI/arXiv identifiers (including old-style, versioned, and URL-prefixed arXiv forms), LaTeX-style `\cite{...}` keys with multi-key splitting, numbered bibliography references, reference-to-citation links for bibliography entries, inline numeric-citation links back to local bibliography nodes, and explicit bibliography `source_url` lifting when numbered reference entries include plain external URLs. It also lifts basic local reference metadata such as year/title/DOI/arXiv when those are present, but it does not yet resolve external paper metadata.
- DOCX documents now get a lightweight synchronous zip/XML pass for title/author/subject/description lifting, heading structure, local filename mention references, inline citation detection, and basic numbered-reference parsing when a references section is detected.
- XLSX spreadsheets now land as document nodes with workbook metadata, sheet-name structure, workbook-to-sheet containment, and shared-string citation detection, but they still do not extract full cell-level semantics or formulas.
- PDF papers now get a lightweight heuristic pass for metadata title/author/subject, `Tj`/`TJ` text extraction, local filename mention references, inline citation detection, and basic numbered-reference parsing when local bibliography lines are recoverable from heuristic text operations, but binary paper/document formats are still not parsed as deeply as the richer text/document formats.
